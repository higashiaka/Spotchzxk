# 정식 출시 로드맵

## 목적

베타 서비스에서 누적된 초대형 자산과 왜곡된 AMM 상태를 정리하고, 구글 연동 계정 및 베타 보상 대상만 보존한 상태로 정식 시즌을 시작한다. 정식 시즌은 초기화된 시장, 새 AMM 티어, 사전 상장 종목, 베타 참여 보상을 기준으로 운영한다.

## 핵심 방향

- 구글 연동 계정 목록은 보존한다.
- 게스트 계정 및 시즌성 데이터는 초기화한다.
- 베타 참여 보상 지급을 위해 보상 대상 스냅샷을 먼저 만든다.
- 정식 초기 자본은 1,000만 코인으로 시작한다.
- 정식 시장은 기존 베타 종목 상태를 이어받지 않는다.
- 정식 신규 상장은 새 AMM reserve 티어를 적용한다.
- 오픈 초기에 시장이 비어 보이지 않도록 100개 내외 종목을 사전 상장한다.
- 추가 100개 내외 종목은 오픈 후 순차 상장한다.

## 보존 대상

정식 출시 초기화 전에 아래 정보는 반드시 스냅샷으로 보존한다.

```text
users.id
users.email
users.display_name
google uid/provider id
chzzk 연동 정보
created_at
last_login_at
베타 참여 여부
베타 보상 지급 여부
```

보상 지급을 위해 별도 테이블을 만든다.

```sql
CREATE TABLE beta_reward_targets AS
SELECT
    id AS user_id,
    email,
    display_name,
    google_uid,
    chzzk_channel_id,
    created_at,
    last_login_at,
    coin_balance,
    NOW() AS snapshotted_at
FROM users
WHERE is_guest = 0;
```

실제 컬럼명은 운영 DB 스키마 기준으로 적용 전에 확인한다.

## 초기화 대상

정식 시즌에서 이어받지 않을 데이터는 초기화한다.

```text
유저 코인 잔고
유저 보유 주식
미체결 주문
거래 내역
종목 및 AMM 풀
배당 내역
시즌 랭킹
시즌 통계
게스트 계정
시즌성 알림/이벤트 진행 상태
```

단, 보상 지급에 필요한 베타 기록은 삭제 전에 반드시 스냅샷으로 남긴다.

## 베타 보상

베타 자산 규모는 정식 경제로 직접 이전하지 않는다.

추천 보상:

- 베타 참여 유저 종목 추가권 10개
- 정식 출시 후 치지직 연동 유저 추가 보상
- 베타 참여 칭호 또는 뱃지
- 상위권 유저 명예 기록
- 소량의 정식 시즌 코인 보상

정식 코인 보상은 초기 자본 1,000만 기준 경제를 흔들지 않는 수준으로 제한한다. 베타 총자산 비례 보상은 적용하지 않는다.

베타 참여 기본 보상은 구글 계정 기준으로 지급한다. 치지직 연동은 베타 보상 수령 조건으로 강제하지 않고, 정식 출시 후 추가 보상과 정식 계정 전환 혜택으로 유도한다.

```text
베타 참여 기본 보상: 구글 계정 기준 지급
치지직 연동 추가 보상: 정식 출시 후 연동 시 지급
정식 주요 기능: 치지직 연동 계정 중심 운영
```

## 계정 정책

정식 출시 후에는 구글 로그인 종료 여부를 별도로 공지하고, 치지직 계정 연동 중심으로 전환한다. 다만 베타 참여 기본 보상은 기존 구글 계정 기준으로 먼저 지급한다.

권장 흐름:

```text
베타 종료 전: 베타 종료 및 정식 전환 일정 안내
정식 전환 전: 구글 연동 계정 목록 스냅샷
정식 시작: 구글 계정 기준 베타 기본 보상 지급
정식 출시 공지: 치지직 연동 혜택 및 정식 계정 전환 안내
정식 이후: 치지직 연동 시 추가 보상 지급 및 제한 해제
추후: 구글 로그인 종료 또는 제한 적용
```

Firebase 로그인 로직은 인증 세션 처리와 기존 유저 매핑에 재활용하되, 정식 서비스의 주 식별자는 치지직 연동 정보로 옮긴다.

정식 시즌 계정 권한은 3단계로 나눈다.

| 계정 유형 | 역할 | 총 보유 주식 제한 | 주요 제한 |
|---|---|---:|---|
| 게스트 | 체험 계정 | 200주 | 종목 추가 불가, 보상 제외, 랭킹 제외 |
| 구글 계정 | 준회원 계정 | 1,000주 | 정식 주요 기능 제한 |
| 치지직 연동 계정 | 정식 계정 | 제한 없음 | 정상 참여 |

게스트 계정은 정식 서비스에서 체험용으로만 유지한다.

```text
게스트 계정
- 총 보유 주식 200주 제한
- 종목 추가 불가
- 배당 수령 제한 또는 제외
- 공식 랭킹 반영 제외
- 베타/이벤트 보상 대상 제외
- 일정 기간 미접속 시 삭제 가능
```

구글 계정은 치지직 연동 전환을 위한 중간 단계로 유지한다.

```text
구글 계정
- 총 보유 주식 1,000주 제한
- 정식 서비스 체험 및 기본 거래 가능
- 종목 추가 기능 제한 가능
- 베타 참여 기본 보상 수령 가능
- 치지직 연동 시 추가 보상 지급 및 제한 해제
- 추후 구글 로그인 종료 또는 제한 가능
```

치지직 연동 계정은 정식 서비스의 기준 계정으로 사용한다.

```text
치지직 연동 계정
- 총 보유 주식 제한 없음
- 종목 추가권 사용 가능
- 배당, 랭킹, 이벤트 정상 참여
- 치지직 연동 추가 보상 수령 가능
```

정식 초기화 후부터 위 제한을 적용한다. 베타 기간에 이미 보유한 주식 수가 제한을 초과한 경우는 정식 초기화로 정리되므로 별도 강제 매도나 회수 정책을 두지 않는다.

## 정식 경제 설정

정식 초기 자본:

```text
1인당 10,000,000 코인
```

상장가 공식:

```text
listingPrice = sqrt(followerCount) * 300
최소 10,000
최대 300,000
1,000원 단위 내림
```

신규 상장 보호:

```text
상장 후 24시간 동안 유저별 200주 매수 제한
heldQty + pendingBuyQty + orderQty <= 200
```

주문 정책:

```text
전량 체결 또는 전량 거절
부분 체결 없음
자동 분할 체결 없음
```

## 정식 AMM 설정

정식 시즌은 베타 운영 중 누적된 초대형 자산과 AMM 풀 괴리를 초기화하고, 새 경제 규모에 맞는 AMM 파라미터로 시작한다. 베타의 V57/V58류 풀 스케일링은 기존 데이터 보정용이며, 정식 시즌의 기본 운영 방식으로 보지 않는다.

기본 전제:

```text
정식 시즌 시작 시 시장 데이터 초기화
초기 자본 1,000만 코인 유지
베타 참여 보상으로 종목 추가권 지급
신규 상장 24시간 초기 보호 장치 유지
AMM reserve 및 feePool은 DECIMAL(65,0) / Java BigInteger 기준 유지
```

상장가 공식은 유지한다.

```text
listingPrice = sqrt(followerCount) * 300
최소 10,000
최대 300,000
1,000원 단위 내림
```

고팔로워 종목은 높은 상장가와 깊은 풀을 함께 가져가고, 저팔로워 종목은 낮은 상장가와 높은 변동성을 유지한다.

정식 시즌 신규 상장 종목은 아래 reserve 티어를 적용한다.

| 티어 | 팔로워 수 | shareReserve |
|---|---:|---:|
| 1 | 2,000 미만 | 8,000 |
| 2 | 20,000 미만 | 12,000 |
| 3 | 200,000 미만 | 18,000 |
| 4 | 1,000,000 미만 | 30,000 |
| 5 | 1,000,000 이상 | 80,000 |

초기 AMM 풀:

```text
coinReserve = listingPrice * shareReserve
shareReserve = tierShareReserve
```

200주 시장가 매수 기준 대략적인 가격 상승률은 다음과 같다.

| 티어 | shareReserve | 200주 매수 상승률 |
|---|---:|---:|
| 1 | 8,000 | 약 5.2% |
| 2 | 12,000 | 약 3.4% |
| 3 | 18,000 | 약 2.3% |
| 4 | 30,000 | 약 1.3% |
| 5 | 80,000 | 약 0.5% |

신규 상장 후 24시간 동안 유저별 초기 매집 제한을 유지한다.

```text
heldQty + pendingBuyQty + orderQty <= 200
```

이 제한은 가격 급등 방지보다 초기 독점 매집 방지가 목적이다. 가격 충격은 초기 AMM 풀 깊이와 슬리피지 가드로 제어한다.

정식 시즌에서는 자동 리밸런싱을 사용하지 않는다.

제거 이유:

- 현재가를 유지하면서 `coinReserve`를 새로 주입하면 출구 유동성을 인위적으로 생성할 수 있다.
- 펌핑 후 리밸런싱을 이용한 매도 전략이 생길 수 있다.
- 정식 경제에서는 풀 깊이 문제를 초기 reserve와 액면분할로 해결하는 편이 더 중립적이다.

정식 운영 원칙:

```text
자동 리밸런싱 없음
관리자 임의 풀 주입 없음
가격 상승 후 풀 깊이 증가는 액면분할로 처리
```

액면분할은 장기적인 풀 깊이 안정장치로 유지한다.

```text
currentPrice / ratio
basePrice / ratio
listingPrice / ratio
totalSupply * ratio
issuedShares * ratio
preStreamFloat * ratio
user_shares.quantity * ratio
user_shares.preStreamQuantity * ratio
user_shares.avgPrice / ratio
shareReserve * ratio
coinReserve 유지
```

이 방식은 AMM 풀에 새 코인을 만들지 않고 거래 단위만 쪼개므로 경제적으로 중립적이다.

주문 처리는 단순하게 유지한다.

```text
시장가/지정가 모두 전량 체결 또는 전량 거절
부분 체결 없음
자동 분할 체결 없음
```

복잡한 처리는 주문 전 예상 계산과 슬리피지 가드로 옮긴다.

시장가 주문:

- 프론트에서 AMM 예상 체결액과 평균단가 계산
- 백엔드에서 실제 AMM 계산 후 `maxCoinIn` / `minCoinOut` 검증
- 가드 초과 시 전량 거절

지정가 주문:

- 지정가가 가격 한도 역할을 한다.
- 체결 시점에 DB의 최신 AMM 풀로 재계산한다.
- 지정가 조건을 벗어나면 체결하지 않는다.

AMM 관련 숫자는 큰 자산과 reserve를 안전하게 처리하기 위해 확장 타입을 유지한다.

DB:

```sql
coin_reserve  DECIMAL(65,0)
share_reserve DECIMAL(65,0)
fee_pool      DECIMAL(65,0)
```

백엔드:

```text
Stock.coinReserve  BigInteger
Stock.shareReserve BigInteger
Stock.feePool      BigInteger
AmmCalculator      BigInteger 기반 계산
```

프론트:

```text
coinReserve/shareReserve는 JSON 문자열로 수신
주문 예상 계산은 BigInt 기반
```

JS `number`로 reserve를 파싱하면 큰 값에서 정밀도가 깨지므로 reserve는 문자열로 유지한다.

베타 운영과 정식 운영은 분리한다.

베타 운영:

- 기존 자산이 매우 커져 일부 종목의 `issuedShares / shareReserve` 괴리가 극단적으로 커짐
- V57 및 필요 시 V58로 기존 풀을 선별 스케일링
- 목적은 베타 유저의 거래 불편 완화

정식 운영:

- 시장 초기화 후 새 티어 reserve로 시작
- 기존 베타 괴리 데이터를 끌고 가지 않음
- V57/V58식 보정은 정식 기본 운영 정책이 아님

정식 추천 코드 값:

```java
public static long calcTierShareReserve(long followerCount) {
    if (followerCount < 2_000)     return 8_000L;
    if (followerCount < 20_000)    return 12_000L;
    if (followerCount < 200_000)   return 18_000L;
    if (followerCount < 1_000_000) return 30_000L;
    return 80_000L;
}
```


## 사전 상장 계획

정식 오픈 직후 시장이 비어 보이지 않도록 운영자가 사전 상장 종목을 준비한다.

추천 수량:

```text
오픈 전 사전 상장: 100개 내외
오픈 후 순차 상장: 100개 내외
```

선정 기준:

- 팔로워 수가 충분한 채널
- 실제 활동성이 있는 채널
- 유저들이 알아볼 가능성이 높은 채널
- 특정 카테고리에 과도하게 쏠리지 않도록 분산

운영 일정:

```text
D-1: 100개 내외 사전 상장
D+1 ~ D+7: 매일 10~20개 순차 상장
유저 종목 추가권: 운영자 사전 상장 외 롱테일 종목에 사용
```

## 출시 후 모니터링

정식 오픈 직후에는 아래 지표를 집중 확인한다.

## 예상 작업 시간

개발 및 문서 준비:

```text
반나절 ~ 1일
```

운영 적용:

```text
백업, 리허설, 점검 포함 1일 권장
```

초기화 작업은 쿼리 실행 자체보다 검증과 복구 가능성이 중요하므로, 운영 DB 적용 전 반드시 백업과 리허설을 먼저 진행한다.
